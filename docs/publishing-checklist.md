# Publishing Checklist

Use this checklist before you make the repo public and start posting about it.

## Repo Readiness
- confirm the repo name, description, and README title match your public branding
- make sure the first screen of the README explains what the repo is and how to use it
- verify the main skill path and first example both work
- make sure `LICENSE`, `CONTRIBUTING.md`, `SECURITY.md`, and `CODE_OF_CONDUCT.md` are present
- remove any wording that makes the repo sound like a source mirror

## Content Readiness
- read the README from top to bottom like a first-time visitor
- check that the copyable prompt still matches the skill output requirements
- make sure the marketing-agent example is the first example a visitor should open
- make sure docs use plain language and do not assume too much prior agent knowledge

## Launch Readiness
- set the GitHub repo description using the short description in `assets/launch-copy.md`
- pin the most useful files in the README or repo sidebar if your platform supports it
- prepare one Twitter/X post, one LinkedIn post, and one Reddit post from `assets/launch-copy.md`
- be ready to answer one practical question: “How do I use this in my own project?”

## Good First Follow-Ups
- add one more worked example after launch
- collect common user questions and turn them into docs improvements
- tighten the skill based on real outputs from coding agents
